# 成绩分析模块融合方案草案

> 基于好分数深度调研 + edu-cloud 现状分析，提出融合方案供 Claude×GPT 讨论

## 一、两平台能力对比

### 功能覆盖矩阵

| 功能域 | 好分数 | edu-cloud | 差距 |
|--------|--------|-----------|------|
| **考后报告（4Tab）** | 总览/科目/班级对比/学生排名 | 同结构 4Tab | 基本持平 |
| **分数分布** | 柱图 + 分段颜色自定义 | 柱图 + 分段配置 CRUD | 持平 |
| **班级对比** | 双轴柱图 + 彩色标签 | 双轴柱图 + 箱线图(后端有前端未接) | edu 后端超前 |
| **学生排名** | 搜索 + 分页 + CSV | 搜索 + 动态科目列 + PDF/XLSX | edu 更强 |
| **成绩趋势** | 无独立页面 | 独立页面(年级/班级/学生+对比模式) | edu 更强 |
| **知识点诊断** | 三维面板(难度/覆盖面/方差)+掌握率表 | class_knowledge+layer_service(前端未充分展示) | 好分数展示优 |
| **学生个人画像** | KP雷达+三线进度条+趋势折线 | StudentAnalysis 有数据(展示薄弱) | 好分数展示优 |
| **分层学情** | 4层可配+堆叠柱图+趋势+名单 | layer_service 后端有(前端无页面) | 好分数有 edu 半成品 |
| **对比分析** | 班级/年级/考试三模式+CV%+雷达 | 无独立对比页 | 好分数有 edu 无 |
| **自定义报告** | 维度选择器+模板保存+图表切换 | 无 | 好分数有 edu 无 |
| **等级赋分** | ABCDE配置+前后对比+CSV | 无 | 好分数有 edu 无 |
| **报告导出** | CSV | PDF+XLSX(但无图表嵌入) | 各有优劣 |
| **级联筛选** | powerOptions 一次返回全树 | 多步级联 | 好分数更优 |
| **AI 诊断** | 无 | 模板拼接(非LLM) | 都弱 |
| **AI 阅卷质量报告** | 无 | 7区块完整实现(置信度/偏差/审计) | edu 独有 |
| **BKT 知识追踪** | 无(模拟数据) | StudentKnpMastery(上游模型) | edu 独有 |

### 核心差距总结

**好分数优于 edu-cloud 的：**
1. 知识点展示密度（三维诊断面板 vs 数据积压在后端）
2. 学生个人画像（KP 雷达 + 三线进度条）
3. 分层学情完整 UI
4. 对比分析独立页面（多模式 + CV%）
5. 级联筛选 UX（powerOptions 一次加载）
6. 自定义报告 + 模板保存

**edu-cloud 优于好分数的：**
1. AI 阅卷质量报告（7 区块，好分数完全没有）
2. BKT 知识追踪数据（贝叶斯模型 vs 模拟数据）
3. 成绩趋势独立页（对比模式 + 双 Y 轴）
4. PDF/XLSX 导出能力
5. 有效分优先级链（confirmed > ai_done > human > scan）

## 二、融合定位

### 产品定位公式

> **好分数的统计深度 + edu-cloud 的 AI 能力 = 差异化竞争力**

好分数是"统计分析平台"——给你数据，你自己看。
edu-cloud 要做"AI 分析平台"——给你数据，还告诉你意味着什么、应该怎么做。

### 三层架构

```
┌─────────────────────────────────────────────────┐
│  Layer 3: AI 洞察层（edu-cloud 独有优势）         │
│  LLM 诊断 / 教学建议 / 成长预测 / 异常预警        │
├─────────────────────────────────────────────────┤
│  Layer 2: 高级分析（借鉴好分数）                   │
│  KP 三维诊断 / 分层学情 / 对比分析 / 等级赋分      │
├─────────────────────────────────────────────────┤
│  Layer 1: 基础统计（对标好分数，已基本完成）         │
│  考后报告 / 分布 / 排名 / 趋势 / 导出              │
└─────────────────────────────────────────────────┘
```

## 三、具体融合方案

### Phase 2: 借鉴好分数展示层（T3，~3天）

重点借鉴好分数的信息密度和交互模式：

| 功能 | 借鉴点 | 实现方式 |
|------|--------|---------|
| KP 三维诊断面板 | 难度/覆盖面/方差三视角 | AnalyticsPage 新增 Tab 或卡片 |

| 对比分析页 | 班级/考次对比 + CV% | 新页面 |

### Phase 3: AI 洞察层（T3-T4，核心差异化，~5天）

这是 edu-cloud 相对好分数的核心竞争力：

| AI 功能 | 输入数据 | 输出 | 实现方式 |
|---------|----------|------|---------|
| **考试智能诊断** | 分布+KP掌握率+错题模式 | 结构化诊断报告(问题→原因→建议) | LLM(gemini) + prompt template |
| **班级对比洞察** | 多班统计数据+CV% | "哪个班在哪科明显偏弱，可能原因是..." | LLM 生成 |
| **学生成长分析** | 历次考试趋势+KP变化 | 个性化学习路径建议 | LLM + BKT 数据 |
| **教学建议** | 班级 KP 薄弱点+错题模式 | "建议在X知识点增加Y类型练习" | LLM 生成 |
| **异常预警** | 成绩骤降/KP突变 | 自动标记+预警通知 | 规则触发 + LLM 解读 |
| **AI 综合报告** | 全量数据 | 可打印的 AI 分析报告(含图表) | LLM 生成叙述 + ECharts 图 |

## 四、Claude×GPT 共识（2026-05-08 两轮讨论）

### 决策 1: Phase 优先级 → 交叉推进

**共识**：不按 Phase 顺序串行，按以下节奏交叉推进：

| 顺序 | 功能块 | 说明 |
|------|--------|------|
| 0 | 筛选与事实契约 | 定义统一筛选、score policy、snapshot_hash、fact_id；AI 和页面复用前提 |
| 1 | 成绩分析工作台 | 侧栏不扩张，做页面内 Tab、共享筛选、默认最近考试 |
| 2 | 激活存量接口 | 接上 classBoxplot、classKnowledge、layerAnalysis、trend、export |
| 3 | 好分数式展示 MVP | 分数段、班级对比、知识点短板、分层学情四块，先不做模板系统 |
| 4 | AI 班级诊断 MVP | 基于同一 snapshot 异步生成，先覆盖"班级+学科+考试" |
| 5 | AI 学生成长/BKT | 接跨考试知识点趋势、分层建议、学生跟进清单 |
| 6 | 自定义报告/等级赋分 | 等指标块稳定后再做 |

**可并行**：前端工作台 & 后端 snapshot 契约；P2 各展示 Tab；AI prompt/eval fixture（snapshot 字段确定后）
**不可并行**：AI 正式入口不早于 score policy 和权限快照；自定义模板不早于指标块注册

### 决策 2: AI 诊断数据输入 → 混合策略（预聚合快照为主）

**共识**：以预聚合快照为主，实时聚合为辅。

- 缓存 key 含 `exam_id + school_id + scope + subject + class + filters + score_version + knowledge_version`
- LLM 只吃压缩后的结构化事实，不直接扫原始明细
- 同一考试的 AI 报告必须可复现（教研会、家长会、归档场景）

**AI Prompt 输入 Schema（班级诊断示例）**：

```jsonc
{
  "schema_version": "analytics_class_diagnosis.v1",
  "context": {
    "school_scope": "current_user_school",
    "exam_id": 123, "exam_name": "高一上期中",
    "grade_id": 10, "class_id": 3, "subject_code": "math",
    "user_role": "homeroom_teacher"
  },
  "snapshot": {
    "snapshot_hash": "score_v17_filter_a91c",
    "score_policy": "confirmed > ai_done > human > scan"
  },
  "data_quality": {
    "student_count": 45, "scored_count": 44,
    "knowledge_mapping_coverage": 0.86,
    "low_evidence_concepts": ["K102"]
  },
  "score_summary": {
    "full_score": 150, "avg": 101.2, "median": 103, "stddev": 18.4,
    "pass_rate": 0.78, "excellent_rate": 0.16,
    "segments": [{"label": "90-120", "count": 21}]
  },
  "comparison": {
    "grade_avg": 96.5, "class_avg_delta": 4.7,
    "class_rank_in_grade": 2, "parallel_class_count": 8,
    "previous_exam_avg_delta": -2.1
  },
  "knowledge_points": [
    {
      "fact_id": "kp:K031", "name": "二次函数最值",
      "class_rate": 0.52, "grade_rate": 0.61, "gap_to_grade": -0.09,
      "unmastered_student_count": 19, "evidence_level": "medium"
    }
  ],
  "student_groups": [
    {
      "fact_id": "layer:need_help", "label": "待提升",
      "count": 11, "avg": 62.4, "weak_concepts": ["K031", "K044"]
    }
  ],
  "constraints": {
    "max_findings": 5, "max_actions": 6,
    "forbid_unbacked_claims": true, "hide_student_names": true
  }
}
```

**AI 输出也结构化**：`summary`、`findings[]`、`risk_alerts[]`、`teaching_actions[]`、`student_followups[]`、`data_limits[]`，每项带 `evidence_fact_ids` 和 `confidence`。

### 决策 3: AI 报告缓存 → 数据版本失效 + 软 TTL + 手动重生成

**共识**：不单纯靠 TTL。

- 缓存 key 含 `snapshot_hash、prompt_version、model_version、filters、role_scope`
- 考试类报告：7 天软 TTL（考试定稿后数据不变）
- 学生成长类（含近期考试）：24-72 小时
- 提供"重新生成"按钮（手动刷新）

### 决策 4: 自定义报告 → 不进 P1/P2，先做章节开关

**共识**：自定义报告的模板系统需要指标块注册、权限、版本化渲染、导出一致性，过早泛化风险大。先做固定综合报告 + 章节开关（教师可隐藏/展示不同分析板块），模板保存放后续。

### 决策 5: 等级赋分 → 纳入但作为可选专项

**共识**：纳入（已有 `level_score_service.py` 基础），但作为可选专项能力，不进默认主路径。需补：政策配置、并列分处理、审计记录、导出标识。**赋分只能是派生视图，不能覆盖有效分链路。**

### 决策 6: 页面结构 → 统一入口 + 页面内 Tab

**共识**：

- **侧栏保持 2 个入口**："成绩分析" + "AI 阅卷报告"（建议改名"阅卷质量报告"，避免与 AI 综合报告混淆）
- **"成绩分析"页面内 6 个 Tab**：考后总览、班级/年级对比、知识点诊断、学生/分层学情、趋势追踪、AI 综合报告
- **Tab 复用同一筛选上下文**（选考试→科目→班级→各 Tab 共享），避免 6 个独立页面重复筛选/权限/空态逻辑
- 现有路由做子路由或重定向，不破坏已有链接

### GPT 额外风险提示

1. **好分数 clone 的 study.js 知识点数据是随机模拟**，适合借鉴信息架构，不能作为计算口径
2. **多学校、多角色场景**：筛选树必须全程基于 ID 和权限裁剪，不能走好分数的 class name fallback
3. **AI 输出必须拆成"事实、推断、建议"**，并显示数据覆盖率；KP 映射不足时降级为"数据不足"
4. **统一有效分链路**：等级赋分、AI 报告、导出报表不能各自使用不同 score fallback
5. **BKT / StudentKnpMastery 是差异化机会**：做"跨考试成长诊断"，输出持续薄弱/最近改善/新增风险/教师跟进动作

### GPT 对 BKT 数据利用的建议

- **班级报告**：输出"知识点风险地图"——KP 名、班级掌握率、年级掌握率、差值、影响人数、关联题目、补救建议
- **学生报告**：输出"成长画像"——持续薄弱 Top3、最近改善 Top3、新增风险点、练习方向、教师跟进
- **分层报告**：结合 layer_service 分层结果，给优秀/稳定/待提升层分别生成教学策略
- **低置信度边界**：低题量 / 低 KP 映射覆盖率 / 单次考试 → 不能下"能力稳定缺陷"结论，只能标低置信度风险

### GPT 对 powerOptions 适配的建议

- MVP 保留全树（已有 `power_options_service.py`）
- 正式多校多角色场景改为权限内、懒加载、版本化：加 `depth / parent_id / year / exam_type / limit / cursor`
- 学校范围从 token 推导，普通用户不能传 school_id 越权
- 缓存 key：`school_id + role_scope_hash + year + exam_type + depth + parent + data_version`

## 五、约束与风险

- edu-cloud 的有效分优先级链（confirmed > ai_done > human > scan）是数据可信度基础，融合时不可破坏
- AI 诊断成本需要评估（gemini token 消耗 × 调用频次）
- 好分数的 powerOptions 模式假设数据量可控（全量级联树一次返回），edu-cloud 正式版需懒加载
- 当前 AI 阅卷质量报告（7 区块）已完整，是独有优势，融合时保留独立入口
- 好分数 study.js 知识点数据是随机模拟，只借鉴信息架构，不借鉴计算逻辑
