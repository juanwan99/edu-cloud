## 总体判断

优先选择"低迁移成本、强隔离、可渐进演进"的路线，不要为了概念上的"平台化"过早引入过多服务拆分或过度灵活的权限体系。

## Q1: 单体 vs 微服务

**判断：B — 保持分离，前端统一壳 + BFF 聚合**

理由：`exam-ai` 已有生产用户且"不可大改"，直接合并成单体会把最稳定的资产拖入高风险重构；而独立 `API Gateway` 方案对"个人开发者 + 单机部署"来说又偏重。`BFF` 是当前最合适的折中点，既能统一三栏工作台的上下文、鉴权、聚合查询和 AI 会话入口，又不需要重写现有后端边界。

风险提示：`BFF` 很容易演变成"第二个单体后端"，把业务、聚合、权限、AI 编排都堆进去。要从一开始限定它只做聚合、会话上下文和前端适配，不承载领域核心逻辑。

## Q2: 前端架构选型

**判断：A — Vue 3 + Naive UI**

理由：这是最低迁移成本的选择，能直接复用 `exam-ai` 现有的 `ChatPanel`、`DashboardLayout`、`AnalyticsPage` 及已有认知模型。三栏 AI 工作台本质上是复杂状态驱动界面，不适合原生 HTML/JS；切到 React + Ant Design 会带来整套技术栈重学、组件重写和样式体系重建，对单人开发不划算。

风险提示：`Naive UI` 的生态和企业模板沉淀不如 Ant Design 丰富，后期复杂表格、低代码配置、可视化编辑器等能力可能不够顺手。应尽早抽出设计 token、布局骨架和通用业务组件，避免未来被组件库绑死。

## Q3: AI Agent 部署位置

**判断：C — 独立服务，但采用"渐进抽离"**

理由：从平台目标看，AI Agent 已经不是 `exam-ai` 的附属能力，而是整个学校操作系统的核心执行层。放在 `edu-cloud` 会让校本场景反向依赖云端；继续放在 `exam-ai` 又会让新平台受制于旧系统。独立服务最利于复用、统一接入 `llm-proxy`、统一工具治理、统一审计和后续扩展到 Studio/通知/知识库。

风险提示：过早抽服务会带来接口重构、工具注册迁移、回归测试和运维复杂度。如果现阶段资源紧，建议先保留 `exam-ai` 内已有引擎运行，同时把工具协议、会话模型和调用接口先抽象出来，再逐步迁到独立服务。

## Q4: 数据融合方案

**判断：C — 混合：高频同步 + 低频联邦查询**

理由：`A` 的全量同步在教育场景里会放大隐私、同步一致性和存储成本问题；`B` 的纯联邦查询又会让跨校分析、趋势聚合和 AI 响应速度变差。混合方案最符合业务：把高频分析所需的事实数据和维表沉到云端，保证仪表盘和跨校比较性能；把低频、明细、强敏感数据按需回源查询，降低复制面。

风险提示：混合方案最怕"边界不清"，最后变成双写和双真相。必须明确每类实体的 `source of truth`、同步频率、容忍延迟、冲突处理和数据新鲜度标记。

## Q5: 家校通信渠道

**判断：C — 企业微信为主 + 短信降级（但架构上首期按 A 实现）**

理由：企业微信是事实主渠道，低成本、触达自然、与学校组织关系匹配，适合通知审批流；但只依赖企业微信会在"紧急通知、未读升级、家长未绑定"等情况下失效。短信不应该作为主通道，而应该只覆盖关键告警、强时效事项和发送失败补偿。

风险提示：短信一旦引入，就会带来成本控制、模板报备、发送合规、供应商 SLA 和失败重试策略，不能临时拼接。要从一开始定义"什么级别的消息能触发短信"。

## Q6: 三栏布局的响应式策略

**判断：A — 桌面端三栏，移动端 Tab/抽屉切换**

理由：学校管理与分析场景的主生产力终端仍然是桌面端，三栏是核心体验；但审批、查看结果、发通知这些动作确实需要移动可用。只做桌面端会损失大量真实使用场景；单独做小程序则相当于再维护一套前端。`PWA` 本身不是信息架构策略，只是交付形态，不能替代移动交互设计。

风险提示：如果只是简单把三栏压缩成三个 Tab，移动端体验会很差。需要明确移动端只保留高频任务流，比如"待审批""查看班级画像""发送通知""查看 AI 草稿"，而不是强行完整复刻桌面工作台。

## Q7: RBAC 粒度

**判断：A — 7+ 显式角色 + scope，底层预留向 C 演进**

理由：这个平台是"角色驱动一切"，不同角色不仅权限不同，连默认上下文、首页工作台、可见 AI 工具和审批职责都不同，仅靠粗粒度角色 + scope 不足以支撑产品表达。校长、教务、班主任、教师、家长、教育局这些角色需要在产品层面是显式存在的。

风险提示：如果把角色写死在代码里，后面会出现"一个人多个角色""代班/兼课/跨校""临时授权"等复杂情况。建议底层模型采用 `role + permission + scope`，前台先提供固定角色模板，不要一开始就开放完全自定义组合。

## 简报中遗漏的关键架构决策（8 项）

### 1. 多租户模型
是"全平台单库多租户 + RLS"，还是"按学校独立 schema / 独立库"，这是最核心的隔离决策之一，直接影响权限、备份、运维、性能和合规。

### 2. 身份认证与账号统一
教师、校领导、教育局、家长是否共用一套身份系统；是否以企业微信作为主登录；家长账号如何绑定学生；一个人多角色如何切换，这些不先定，后面 RBAC 会反复返工。

### 3. 数据同步与事件机制
`exam-ai` 到 `edu-cloud` 的同步是 API 拉取、定时任务、CDC、消息队列还是 Outbox 事件。没有这一层，Q4 的"混合融合"落不了地。

### 4. AI 工具治理和安全边界
不同角色可调用哪些工具，哪些动作必须人工审批，AI 是否允许直接发通知/改数据/触发外部接口，工具调用日志如何审计，这决定平台是否可控。AI 在这里不是插件，而是主界面入口，所以没有 trace、审计和成本治理，这个平台很快会失控。

### 5. Studio 文档存储与版本策略
AI 草稿、人工修改稿、审批稿、最终发送稿是否版本化，存数据库还是对象存储，是否支持 PDF/Word 导出，这会影响右栏"行动队列"的真正可用性。

### 6. 事件流与任务编排模型
你已经有"日历事件 → AI 拟稿 → 审批 → 发送"的设想，这本质上不是普通 CRUD，而是工作流系统。需要明确：
1. 是否引入统一任务状态机。
2. 是否采用 outbox/event log 保证通知、同步、审批动作可追踪。
3. 长任务如何异步执行，如何回写结果到工作台。

如果没有事件和任务模型，后续审批、文档生成、消息发送会彼此耦合，问题会集中出现在"重试后重复发送"和"状态不一致"上。

### 7. 安全与合规边界
教育数据高度敏感，简报里还缺少：
1. 学生隐私数据分级。
2. 脱敏、加密、审计日志、保留周期。
3. AI 输出是否允许直接用于家长通知、学生评价，是否需要强制人工复核。

这部分不是法务附属问题，而是直接影响数据模型、日志策略和产品流程的架构问题。

### 8. 知识库接入模式
`edu-knowledge-base` 是预处理成检索索引、向量库、标签库，还是继续以 JSON 直读；校本文档是否也进入同一检索体系；RAG 与结构化数据查询如何路由，这会影响 AI 回答质量和成本。

## GPT 建议的落地顺序

如果只给一个落地顺序建议：

1. **先做 Q1+B / Q2+A / Q4+C / Q7=A** 作为平台骨架
2. **再推进 Q3=C** 的 Agent 抽离
3. **最后补 Q5 和 Q6** 的渠道与移动端闭环

这样最符合"单人开发、已有系统不可大改、但产品目标要持续放大"的现实约束。
