## F002 Round 3: 关闭剩余 2 条越权面

### root_cause

Round 2 的 `check_class_scope` + `check_resource_class` 只覆盖了以 `{resource_id}` 作为**路径参数**的端点。但有两类写入路径通过 **request body 字段**传入外班 ID，绕过了 resource-class 校验：

1. `POST /classes/{class_id}/records` + `/records/batch` (`add_points`) — body 含 `rule_item_id`（可选），service 直接写入 `conduct_records.rule_item_id`，未校验 `rule_item_id` 所属 class
2. `POST /classes/{class_id}/groups/{group_id}/members` (`add_group_members`) + `DELETE .../members/{student_id}` (`remove_group_member`) — body/path 含 `student_ids`/`student_id`，未校验 student 是否属于本 class

### preserved_invariants

- ORC-001 ~ ORC-004（R2 已建立，保持）
- **ORC-009（新）**：`conduct_records.rule_item_id` 必须满足 `rule_item.category.class_id == record.class_id`（外键级关联一致性）
- **ORC-010（新）**：`conduct_group_members` 写入时 `student.class_id == group.class_id`（组员与组归属一致）
- **ORC-011（新）**：scope 校验发生在 service 层**之前**（router），service 不重复校验但也不绕过

### non_goals

- **不**改动现有 `check_class_scope` / `check_resource_class` 语义（已通过 R2）
- **不**新增路由层级（继续用 `/classes/{class_id}/...`）
- **不**改 ORM 表结构（8 表冻结，Alembic 只增不改）
- **不**改写 service 层的业务逻辑（仅在 router 前置校验新 invariant）
- **不**引入 WHERE class_id 过滤作为替代——校验必须在写入前明确 raise，不能让外班数据被静默忽略（静默忽略=软性 success 违反越权检测语义）

### allowed_change_surface

- `src/edu_cloud/modules/conduct/permissions.py`:
  - 新增 `check_rule_item_class(db, rule_item_id, class_id)` — 通过 join `rule_categories` 验证 rule_item 属 class
  - 新增 `check_students_class(db, student_ids, class_id)` — 批量验证 students 都属 class（单条失败整体拒绝）
- `src/edu_cloud/modules/conduct/admin_router.py`:
  - `add_points` / `add_points_batch`（line 93-128）：如 `data.rule_item_id is not None` → `await check_rule_item_class(...)`
  - `add_group_members`（line 326-337）：`await check_students_class(db, data.student_ids, class_id)`
  - `remove_group_member`（line 340-351）：`await check_students_class(db, [student_id], class_id)`
- `tests/test_conduct/test_admin_api.py` — 补 3 条越权红测

## N001: 恢复 id_card 后 6 位契约

### root_cause

Round 2 修复 F005 时，phone 分支正确走了 Option A (`profile.verify_code`)，但**同步重写了 id_card 分支**（line 188-191）：

```python
if verify_type == "id_card":
    stored = decrypt(profile.id_card_number) if profile else None
    if not stored or stored != verify_code:     # ❌ 整串相等
        raise ValidationError("身份证号验证失败")
```

同时新增的 `test_parent_bind_id_card_mode` 使用整串身份证号 `110101200001011234` 作为 `verify_code`，把"整串相等"这个未授权行为**固化到测试**里。

Round 1 F005 用户决策明确 Option A：`id_card` 仍独立走"身份证号**后 6 位**比对"。Round 2 违反该决策。

### preserved_invariants

- **ORC-012**：id_card 验证比对 `decrypt(profile.id_card_number)[-6:] == verify_code`（后 6 位）
- **ORC-013**：phone/custom 共享 `profile.verify_code` 路径（Option A R2 已建立，保持）
- **ORC-014**：`profile.id_card_number` 入站必经 AES-256-GCM 加密（不因本次修改而降级）
- **ORC-015**：verify_code 输入长度约束：id_card 模式下 len=6（纯数字或末位 X）；phone/custom 无固定长度

### non_goals

- **不**改 phone/custom 路径（Option A 已生效）
- **不**改 `profile.id_card_number` 字段语义 / 加密方式
- **不**在入口加「后 6 位截取」—— 用户提供的 `verify_code` 就是后 6 位字符串（不做服务端切片错位）
- **不**容忍测试修改去迎合错误实现（L017 红旗反模式）—— 必须把实现对齐契约，同时修测试对齐契约
- **不**添加"后 6 位 OR 整串"双路径（契约单一：仅后 6 位）

### allowed_change_surface

- `src/edu_cloud/modules/conduct/parent_service.py` (line 188-191):
  ```python
  if verify_type == "id_card":
      stored = decrypt(profile.id_card_number) if profile else None
      if not stored or stored[-6:] != verify_code:    # 恢复：比对后 6 位
          raise ValidationError("身份证号验证失败")
  ```
- `tests/test_conduct/test_parent_api.py::test_parent_bind_id_card_mode`:
  - fixture 数据不变（id_card_number 仍存完整身份证号）
  - 绑定请求改为 `verify_code=完整身份证号后 6 位`
  - 错误用例：`verify_code=完整身份证号` → 应失败（因不等于后 6 位）
  - 错误用例：`verify_code=错误后 6 位` → 应失败
- `docs/archive/plans/2026-04-12-conduct-module-design.md` §3 契约段追加："id_card 模式比对后 6 位（Option A 锁定）"（防退化 sentinel）

### verification

- **修正测试 T4**：`test_parent_bind_id_card_mode` — 用后 6 位验证成功
  - 反向验证：把实现改回 `stored != verify_code` → 测试失败（因 `verify_code` 是 6 字符，stored 是 18 字符）
- **新增红测 T5**：`test_parent_bind_id_card_full_string_rejected` — 用完整身份证号作 verify_code → 应 403（防 Round 2 行为复活）
- **新增红测 T6**：`test_parent_bind_id_card_wrong_6_digits` — 用错误的后 6 位 → 应 403（反向边界）
- 无行为变更审批：N001 是 R1 已确认契约的**回归**，非新行为变更，不需要 user decision 二次确认（用户已明确 REJECT R2 的"整串相等"新契约）
