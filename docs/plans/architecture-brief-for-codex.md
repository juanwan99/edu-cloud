# edu-cloud 架构重构讨论简报

## 背景

edu-cloud 当前是一个纯后端 API 服务（FastAPI，端口 9000），实现了联考 MVP（58 tests，GPT 3 轮 PASS）。用户希望将其重构为**学校超级智能管理平台**——AI 驱动的全业务操作系统。

## 现有资产

| 子系统 | 技术栈 | 状态 | 说明 |
|--------|--------|------|------|
| exam-ai | FastAPI + Vue 3 + Naive UI | AI Agent Phase 1 ✅ | 学校端：阅卷+AI 分析（23 工具 ReAct 循环）|
| edu-cloud | FastAPI（无前端）| 联考 MVP ✅ | 云端：联考编排+跨校数据汇总 |
| edu-knowledge-base | JSON 数据 | 课标+教材+高考题 ✅ | 知识底座：151 课标+2135 教材块+2291 高考题 |
| paper-skill | Express + 原生 HTML/JS | 9 阶段写作 ✅ | AI 论文写作系统 |

## 用户愿景

参照 Google NotebookLM 三栏布局，构建学校智能管理平台：

```
┌─────────────┬──────────────────────┬─────────────────┐
│   左栏        │      中栏             │     右栏         │
│  工作上下文    │   数据呈现 + AI 对话   │   Studio 产出    │
│             │                      │                 │
│ - 校本数据    │ - 仪表盘/图表/表格     │ - 教学诊断报告   │
│ - 云端数据    │ - 统计卡片            │ - 家长通知单     │
│ - 知识库     │ - 趋势分析            │ - 学期计划      │
│ - 联考数据    │                      │ - 评语/评价     │
│             │ ──────────────────── │ - 论文（接paper） │
│             │   AI 对话区（底部）     │                 │
│             │   教师用自然语言提问    │  审批→发送      │
└─────────────┴──────────────────────┴─────────────────┘
```

## 核心特征

### 1. AI 是操作界面本身（不是附加功能）
- 登录后的主界面就是三栏工作台
- AI 感知用户角色、当前上下文，自主规划分析路径
- AI 可以调用工具查询数据、生成报告、拟通知、出评语

### 2. 角色驱动一切
不同角色看到不同的内容，能做不同的操作：

| 角色 | 左栏（上下文） | 中栏（工作台） | 右栏（产出） |
|------|-------------|---------------|------------|
| 校长 | 全校数据、教师列表 | 办学指标仪表盘 | 学校发展报告、教师考核 |
| 教务主任 | 考试安排、课表 | 排考/教学进度 | 考试安排表、教务通知 |
| 年级组长 | 本年级班级 | 年级横向对比 | 年级分析报告 |
| 班主任 | 本班学生、各科成绩 | 班级画像 | 学生评语、家长通知 |
| 科任教师 | 所教班级、知识点 | 成绩分布/错题 | 教学反思、补差计划、论文 |
| 教育局 | 辖区学校 | 跨校对比 | 区域质量报告 |

### 3. 数据双源融合
- **校本数据**：来自 exam-ai（考试、学生、成绩、行为记录）
- **云端数据**：来自 edu-cloud（联考跨校数据）+ edu-knowledge-base（课标/教材/文献）
- AI 对话时自动根据任务选择数据源

### 4. 家校通信 + 审批流
- 学期日历驱动：预设事件（放假/考试/家长会）→ 自动提前拟通知
- AI 自动生成通知内容 → 负责人审批 → 群发（企业微信 API）
- 权限分级：有权直接发送 / 只能拟稿等审批

### 5. Studio 文档生成
右栏不只是展示，是**待审批的行动队列**：
- AI 生成的报告/通知/评语 → 教师审阅 → 确认/修改 → 执行（存档/打印/发送）
- 类似 NotebookLM 的多种产出格式（报告、图表、思维导图等）

## 已识别的子系统

| # | 子系统 | 建议优先级 |
|---|--------|----------|
| 1 | 三栏 AI 工作台（前端框架） | P0 |
| 2 | RBAC 角色权限 + 数据隔离升级 | P0 |
| 3 | AI Agent 云端化（复用 exam-ai 引擎） | P1 |
| 4 | Studio 文档生成引擎 | P2 |
| 5 | 家校通信（企业微信 + 审批流 + 学期日历） | P3 |
| 6 | 知识库集成（edu-knowledge-base 接入） | P4 |

## 需要 Codex 讨论的架构问题

### Q1: 单体 vs 微服务
当前 exam-ai 和 edu-cloud 是两个独立服务。新平台应该：
- A) 合并为单体（前端统一，后端合并）
- B) 保持分离，前端统一壳 + BFF 层聚合
- C) 保持分离，前端独立部署，API Gateway 统一入口

### Q2: 前端架构选型
- A) Vue 3 + Naive UI（与 exam-ai 现有前端一致）
- B) React + Ant Design（更成熟的企业级组件库）
- C) 原生 HTML/JS（与 paper-skill 一致，无构建步骤）

### Q3: AI Agent 部署位置
- A) AI Agent 引擎在 edu-cloud（云端统一），exam-ai 通过 API 调用
- B) AI Agent 引擎在 exam-ai（保持现状），edu-cloud 通过 API 调用
- C) AI Agent 引擎独立服务（port 8002，Phase 3 设计已有此规划）

### Q4: 数据融合方案
校本数据在 exam-ai（SQLite/PostgreSQL），云端数据在 edu-cloud（PostgreSQL）。
- A) 校本数据全量同步到云端，云端统一查询
- B) 联邦查询：AI Agent 按需调用两个数据源的 API
- C) 混合：高频数据同步到云端，低频数据联邦查询

### Q5: 家校通信渠道
- A) 企业微信 API（免费，微信原生触达）
- B) 自建消息系统 + 短信网关
- C) 企业微信为主 + 短信降级

### Q6: 三栏布局的响应式策略
教师可能用手机/平板访问：
- A) 桌面端三栏，移动端 Tab 切换
- B) 只做桌面端，移动端用企业微信小程序
- C) PWA 自适应

### Q7: RBAC 粒度
当前 edu-cloud 有 4 个粗粒度角色。需要：
- A) 扩展为 7+ 角色（教育局→校长→教务→年级→班主任→教师→家长）+ scope
- B) 保持粗粒度角色，用 scope（学校/年级/班级/学科）细化
- C) 角色模板 + 自定义权限组合

## 约束条件

1. **个人开发者**，无团队，依赖 AI 辅助开发
2. **现有 exam-ai 不可大改**——已有 451 tests + 生产用户
3. **exam-ai 前端是 Vue 3 + Naive UI**，已有 ChatPanel/DashboardLayout/AnalyticsPage
4. **edu-cloud 当前无前端**，从零开始
5. **服务器资源有限**：一台主服务器（SSH server 别名）
6. **LLM 调用走 llm-proxy 网关**（端口 8100，7 provider）

## 外部调研关键发现

1. 科大讯飞是 AI 教育标杆（4.2万校），但硬件绑定+封闭。差异化空间：轻量 SaaS + 开放 AI Agent
2. 企业微信是家校通道事实标准（100% 微信渗透）
3. NotebookLM 三栏布局 = "数据源 | AI 对话 | 结构化产出"，完美映射教育场景
4. 行业共识：混合 UI（管理 GUI + 分析 AI 对话 + 产出 AI 生成）
5. RBAC 最佳实践：角色 + 作用域（scope），PostgreSQL RLS 保证数据隔离
6. Microsoft Copilot 2026 趋势：从"响应命令"到"自主执行"
